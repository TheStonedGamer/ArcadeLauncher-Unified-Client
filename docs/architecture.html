What the platform is

Four cooperating repositories under the TheStonedGamer org, all talking through one central Rust server.

ArcadeLauncher catalogs a personal game library (emulated and PC), lets users install / launch / track games, and layers a full social platform on top — friends, chat, voice, presence, profiles, and game invites.

The server

arcadelauncher-server · port 8721 (REST + WS), admin UI on 8722. A single Rust/axum service backed by MariaDB — the source of truth for catalog, users, social graph, and live session state.

What it exposes

Catalog & games

Catalog metadata, install manifests (Bearer), cover-art.

Auth

Challenge-response + TOTP. Forgot-password emails identical responses to avoid account enumeration.

Social REST + WS

Friends (sortable, sectioned), Chats inbox, DM & room/group chat (edit/delete/react/reply/attach), presence, profiles (XP/level), privacy, game invites, rooms, 1:1 + group-mesh voice relay.

Cloud saves

Per-game save files with traversal rejection & a 50 MB/file cap.

Remote play

Stream an installed game from a paired host. Client plays in-engine via the bundled GPL-separate stream engine (client.start, live status + Stop); external Moonlight as automatic fallback.

Attachments

presign → PUT-to-MinIO → presigned-GET; bytes never touch the DB.

Streaming

TURN credential vending + Sunshine/Moonlight pairing.

Admin UI

TOTP-gated, server-rendered; Cloudflare Tunnel (external) + nginx2 (LAN).

Health / metrics

/api/health reports the live VERSION; /api/metrics scraped by Prometheus.

Deploy & startup. Pushing to main auto-bumps VERSION (bot commit) and builds a release. Prod deploy is git-pull based: pull → cargo build --release → swap binary → systemctl restart. After restart the process runs schema setup + spawns the fs watcher before binding :8721 (~25 s) — health returns connection-refused during that window. Poll ~30 s for API listening on http://0.0.0.0:8721.

The client

Tauri v2 (Rust core) + React + TypeScript. One cross-platform client replacing two retired native codebases — a webview gets rendering, WebRTC, and WebSockets "for free."

Two halves: Rust core + React UI

The golden rule: authed REST never uses fetch — the bearer token stays Rust-side.
The five-file recipe. Wiring a new server endpoint into the client = Rust command → IPC registration → TS api wrapper → React hook → UI, plus tests on both sides.

How a UI action becomes an authed server call and back — the path every feature follows.

Notable subsystems

Social / live gateway

Client useSocial ↔ Rust social WS client ↔ server /ws/social (social_api.rs + fanout.rs). Carries chat, presence, reactions, replies, attachments, profiles, game invites, rooms, and voice signaling. The UI opens on a Chats inbox (active conversations newest-first with unread counts) and the friends roster is sortable (Status / Name / Recent, favorites pinned).

Group rooms & mesh voice. Rooms have their own chat layer (room_chat/room_message frames, per-room log reducer). Group voice runs a full WebRTC mesh — one peer connection per participant — reusing the per-peer voice_signal relay tagged group:true+roomId, routed to a separate handler so 1:1 and group calls coexist on one socket. Peer offer/answer order is coordination-free (lower id offers). Live multi-peer relay is client+protocol-complete; server-side room/call fan-out is the remaining gap.

Heartbeat gotcha. A WS control ping does not wake the receive loop. The client sends an application {"type":"ping"} every 20 s and the server answers with a data-frame {"type":"pong"} that does. Don't remove either side.

Voice — P2P WebRTC

Chosen over a server binary relay. The server only relays voice_signal frames (offer/answer/ICE) — no media passes through it. ICE uses public STUN plus a coturn TURN server (turn.orlandoaio.net) for symmetric-NAT peers. Being P2P, it doesn't interop with the legacy binary-relay client (fine post-cutover; unified↔unified works).

Installer / downloads

Pure FSM core (manifest / paths / sha256 / queue) + live HTTP transport (resumable ranged GET) + zip-slip-guarded extract. The Install button GETs the server manifest (Bearer), resolves app_data/games/<id>, and hands off to the engine.

Cloud saves

Pure plan_sync core decides Upload / Download / InSync / Conflict by mtime; execution does atomic temp+rename and stamps downloads with the server mtime. The GameDetail cloud-saves panel also lists, snapshots, and restores prior save versions (saves_versions / _snapshot / _restore_version).

CI / Release pipeline

CI runs on every push; a signed release fires only on a vX.Y.Z tag.

Three-phase flow avoids the duplicate-release race; the draft publishes only when both legs succeed.

Cutting a client release

1. Land the feature (CI green on both OSes).
2. Add a CHANGELOG.md entry under a new [X.Y.Z] heading
   (the Discord announcer extracts this section verbatim).
3. Bump the version in ALL FOUR files:
     package.json
     src-tauri/tauri.conf.json
     src-tauri/Cargo.toml
     src-tauri/Cargo.lock  (arcade_launcher stanza)
4. Commit "Release vX.Y.Z", then push the vX.Y.Z tag → fires release.yml.

Client-only feature releases are PATCH bumps, to keep major.minor
matching the live server (a minor bump trips version-lockstep).
